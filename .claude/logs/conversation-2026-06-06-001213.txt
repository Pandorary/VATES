  设计文档是蓝图，实际代码在实现时有以下**差异和演进**：
  
  ### 1. 模块映射 — 哪些实现了，哪些合并了
  
  | 设计文档中的模块 | 实际文件 | 差异 |
  |---|---|---|

  | `quotes/tencent.py` | `services/quotes/tencent.py` | ✅ 一致 — 还额外增加了批量接口 `fetch_quotes_batch` |
  | `quotes/sina.py` | `services/quotes/sina.py` | ✅ 一致 |

  ### 2. 实际与设计的关键差异
  
  **数据源替换**：设计用 AkShare 兜底 → 实际用东方财富 API（`EastMoneyProvider`）替代，因为 AkShare 太重且不稳定。
  
  **未实现的模块**：
  - ❌ AkShare 适配器
  - ❌ 独立的 `cache.py` 文件
  - ❌ 资讯采集 `news/collector.py`
  - ❌ 独立的 `storage/repository.py`
  - ❌ `cross_validate.py` 存在但实际用于 `data_engine.py`（LLM 预测链路），不是 quotes 子系统
  
  ### 3. 与 FastAPI 后端的集成方式（核心）
  
  ```
  main.py (lifespan)
    ├─ init_quote_manager()     → 创建 QuoteManager 单例 (TTLCache + FailoverManager)
    └─ start_scheduler()        → APScheduler 每1分钟触发
  
  tasks/quote_jobs.py
    └─ collect_quotes_job()     → 调用 manager.refresh_all_tracked(db)
  
  routers/quotes.py             → REST API /api/v1/quotes/{code}
    └─ 调用 manager.get_quote() → 缓存 → 故障转移 → DB 回退
  
  models/stock.py
    └─ StockQuote 表 (stock_quotes)  → SQLAlchemy ORM 模型
  ```
  
  **三层数据获取链**（`manager.py:33-54`）：
  1. **TTL 内存缓存** (60s TTL, 200条上限) → 命中直接返回
  2. **多源故障转移** (腾讯→东方财富→新浪) → 成功则写缓存 + 入库
  3. **数据库回退** → 所有数据源失败时，读取 `stock_quotes` 表最后记录
  
  **采集范围来源**（`manager.py:149-174`）：动态拉取，而非硬编码50只：
  - 持仓表 `holdings` 中未删除的股票
  - 自选表 `user_watchlist` 中的股票
  - 配置 `STOCK_DEFAULT_CODES` 中默认代码
  
  ### 4. 还有一套并行的数据引擎
  
  `services/data_engine.py` 是为 **LLM 预测链路** 服务的老数据获取模块，它：
  - 自己直接调 HTTP（不经过 quotes 包的缓存/故障转移）
  - 有交叉验证逻辑（`cross_validate_stock` / `cross_validate_industry`）
  - 被 `routers/prediction.py` 和 `routers/chat.py` 使用
  
  注释中已标明"可重构为使用 quotes 包获取行情数据"——这是两套系统并存的状态，尚未统一。
  
  ### 总结
  
  设计文档中的核心架构（**缓存→故障转移→DB**）已完整实现并集成进 FastAPI，但 AkShare 被东方财富替换、资讯采集未实现、独立的缓存和存储模块被内聚到 Manager 中。`data_engine.py` 和 `quotes/` 包是两条并行的数据获取链路，前者为 LLM 预测服务，后者为定时采集 + API 查询服务。
