# 股票数据采集系统 - 最终技术方案（含缓存设计）

## 1. 项目目标（整合版）

开发一个**稳定、免费、可容忍短时滞后**的股票数据采集系统，满足：

| 需求项 | 具体指标 |
|--------|----------|
| **监控范围** | 约 50 只指定股票（代码列表可配置） |
| **行情数据** | 最新价、涨跌幅、成交量、成交额等 |
| **行情最大滞后** | ≤ 1 分钟（可配置，设为 0 则无滞后限制） |
| **资讯数据** | 个股相关新闻 + 行业板块宏观新闻 |
| **数据存储** | SQLite（两张表：`stock_quotes`、`news`） |
| **缓存策略** | 在滞后允许时间内，重复查询同一股票直接返回缓存数据，减少对数据源的请求 |
| **数据源原则** | 免费、可信、稳定；多源自动切换；不使用 Playwright |
| **运行方式** | 定时任务（每分钟采集行情 + 每5分钟采集资讯） |

---

## 2. 整体架构（含缓存层）

```text
[定时触发器] → [调度器] → [数据管理器] 
                              ↓
                    ┌─────────────────────┐
                    │   缓存层 (TTL缓存)   │
                    │   - 按股票代码缓存   │
                    │   - 可配置过期时间   │
                    │   - 命中则直接返回   │
                    └─────────────────────┘
                              ↓ (miss)
                    [多源故障转移引擎]
                              ↓
                    [数据源适配器] → [HTTP API / AkShare]
                              ↓
                    [数据清洗/校验]
                              ↓
                    [SQLite 存储 + 更新缓存]
```

- **缓存层**：使用内存缓存（如 `cachetools` 或自定义 `TTLCache`），避免在配置的时间窗口内重复请求外部数据源。
- **数据管理器**：先查缓存，缓存未命中再调用 `FailoverManager` 获取真实数据，获取成功后更新缓存并写入数据库。
- **调度器**：每分钟触发一次所有股票的采集任务，但缓存机制会确保在 1 分钟内对同一股票的多次请求（例如手动触发或并发）不会重复拉取。

## 4. 缓存设计细节

### 4.1 缓存策略

- **缓存粒度**：按股票代码（如 `600519`）缓存行情数据对象。
- **缓存有效期（TTL）**：可配置参数 `CACHE_TTL_SECONDS`，默认 **60 秒**（与你要求的最大滞后 1 分钟一致）。若设为 0，则禁用缓存（每次都实时请求）。
- **缓存更新时机**：
  - 当通过数据源成功获取新数据后，立即更新缓存（覆盖旧值）。
  - 当某个数据源失败导致无法获取新数据时，**不更新缓存**，继续返回旧缓存（但需记录日志，避免数据源长期失效导致缓存完全过期）。
- **缓存容量**：设置最大条目数（如 200），超过后按 LRU 或 TTL 自动淘汰。

### 4.2 缓存实现方式（推荐）

使用 `cachetools` 库的 `TTLCache`：

```python
from cachetools import TTLCache

# 最大容量 200，TTL 60 秒
cache = TTLCache(maxsize=200, ttl=60)

def get_quote_with_cache(code):
    if code in cache:
        return cache[code]   # 直接返回缓存数据
    # 否则调用多源管理器获取
    data = failover_manager.get_quote(code)
    if data:
        cache[code] = data   # 写入缓存
    return data
```

### 4.3 缓存与滞后的关系

- 你接受 **≤ 1 分钟滞后**，因此设置 TTL = 60 秒意味着：在 60 秒内对同一股票的任何请求都返回上次采集的数据，不会触发新的网络请求。这实际上将数据更新频率限制为 **最多每分钟一次**，完全符合需求。
- 若将 `CACHE_TTL_SECONDS` 设为 0，则每次请求都会穿透缓存实时拉取（滞后最小，但可能增加数据源压力）。

## 5. 数据存储设计（SQLite）

保持不变，两张表如下：

### 5.1 行情表 `stock_quotes`

```sql
CREATE TABLE IF NOT EXISTS stock_quotes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    code TEXT NOT NULL,
    name TEXT,
    price REAL,
    change REAL,
    change_percent REAL,
    volume INTEGER,
    amount INTEGER,
    source TEXT,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_quotes_code ON stock_quotes(code);
CREATE INDEX idx_quotes_time ON stock_quotes(updated_at);
```

### 5.2 资讯表 `news`

```sql
CREATE TABLE IF NOT EXISTS news (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    code TEXT,
    title TEXT NOT NULL,
    url TEXT,
    publish_time TIMESTAMP,
    source_site TEXT,
    content_preview TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_news_code ON news(code);
CREATE UNIQUE INDEX idx_news_unique ON news(title, url);
```

## 6. 调度与缓存配合

- **定时任务**：依然每 1 分钟触发一次 `fetch_all_stocks_quotes()`，该函数内部会遍历 50 只股票，对每只调用 `get_quote_with_cache`。
- **效果**：由于缓存 TTL = 60 秒，第一次执行时会从数据源拉取并缓存；后续若在同一分钟内再次执行（例如手动运行或调度重叠），则全部命中缓存，不会产生任何网络请求。
- **缓存清空**：无需主动清空，TTL 到期自动过期，下一次请求会重新拉取。

## 8. 异常处理与降级（含缓存）

- **缓存穿透保护**：若缓存不存在，且所有数据源均失败，则返回 `None`，不更新缓存，也不写入数据库。
- **数据源切换时**：在 `FailoverManager` 内部记录切换日志。
- **缓存过期后数据源全部失败**：此时 `get_quote` 返回 `None`，调用方应保留旧数据（不做任何操作）还是报错？建议保留上次成功缓存的数据（但需标注已过期）。为简化，可以定义：当缓存过期且所有源失败时，仍返回过期缓存并记录警告。

## 12. 扩展性说明

- 如需增加新的数据源，只需新增适配器类并加入 `FailoverManager` 的优先级列表。
- 如需调整缓存 TTL，修改 `config.CACHE_TTL_SECONDS` 即可，无需改代码。
- 如需支持 Redis 分布式缓存，可将 `cache.py` 替换为 Redis 实现，保持接口不变。
